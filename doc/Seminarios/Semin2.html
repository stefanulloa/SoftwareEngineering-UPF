1.1 Caso de uso: Validar trabajo Subido.

Contexto:

Un editor quiere comprobar que una obra está en la web.

Actor primario:

Editor.

Actores de Soporte:

Ninguno.

Stakeholder implicados:

Melting Port, OurCode.

Precondiciones:

El editor tiene que poder comunicarse con el sistema.

Post Condiciones de éxito:

El editor comprueba que la obra está en la web.

Post Condiciones de fracaso:

El editor comprueba que la obra no está en la web.

Escenario principal de éxito durante la compra: (Basic Flow)

1.El editor tiene que tener acceso a los trabajos de los autores recibidos.

2.El editor pasa el trabajo a un formato estándar.

3.EL editor sube el trabajo en el nuevo formato a la web.

4.La web sube el trabajo.

5.El editor accede a la web como usuario.

6. La web permite navegar por las obras que se encuentran dentro de la web.

7.El editor busca su trabajo.

8.La web se lo muestra(la obra está disponible en la web).

9.El editor da el visto bueno a la obra en la web (la obra es accesible para los clientes).

Extensiones:

4.a. La web no sube el trabajo porque no está en formato estándar.

4.a.1 Se le notifica al editor y se vuelve al paso 2.

8.a. La web no muestra el trabajo cuando el editor la busca.

8.a.1. El editor al observar que no está, vuelve al paso 3 para volver a subir el trabajo.

1.2 Caso de uso: Comprar un Libro.

Contexto:

El usuario quiere comprar un libro.

Actores primarios:

Usuario (generaliza a Autor, Cliente y Administrador).

Actores de Soporte:

Ninguno

Stakeholder implicados:

Administrador, programador.

Precondiciones:

El usuario tiene que estar registrado y validado en el sistema (para poder acceder al servicio de compras).

Post Condiciones de éxito:

El usuario compra el libro.

Post Condiciones de fracaso:

El usuario no consigue comprar el libro.

Escenario principal de éxito durante la compra: (Basic Flow)

1.El usuario indica que quiere comprar un libro.

2.El sistema pide al usuario el tipo de libro que quiere comprar (de acción, terror, novela, etc).

3.El usuario elige el tipo de libro que quiere comprar.

4.El sistema verifica el stock del tipo de libro escogido por el usuario.

5.El sistema pide los datos necesarios para realizar la compra.

6.El usuario introduce los datos pedidos.

7.El sistema verifica que los datos introducidos por el usuario sean correctos.

8.El sistema devuelve la confirmación.

9.El sistema solicita los datos bancarios.

10.EL usuario introduce los datos bancarios.

11.El sistema verifica que los datos bancarios sean correctos y permite realizar la compra del libro.

Extensiones:

7.a. Si algun dato no es correcto.

7.a.1.Volvemos al paso 5.

10.a.Si algún dato bancario es incorrecto.

10.a.1 Volvemos al paso 9.

4. Glosario

OurCode: Empresa encargada del proyecto. Esto incluye todo tipo de profesionales (programadores, diseñadores, etc.)

MeltingPot: Trabajadores afectados por la web, quienes trabajan con ella, y directivos que reciben análisis de los resultados de la web y ordenan cambios sobre esta.

Clientes: Usuarios que deberían de poder usar la web fácilmente.

Autores: Profesionales que deberían de quedar satisfechos con las presentaciones de sus obras en la web, así como el manejo de esta misma.

Paypal: Compañía encargada de tramitar las donaciones de los usuarios.

Servicios (pago, mensajería): [Es un punto oscuro, puesto que no se ha especificado nada sobre estos en el texto].

Formatos de edición de texto (PDF, Latex, Word, etc.): Herramientas útiles para que los autores escriban y envíen su trabajo a la web.

Métodos de adquisición de obras: Esto incluye las opciones de compra, como descarga de formato on-line o envío de formato físico.
